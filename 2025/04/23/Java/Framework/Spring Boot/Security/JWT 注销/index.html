JWT 注销

本文最后更新于 2025年7月4日 晚上

先掌握 Spring Boot 3 + Spring Security 6 - 使用 JWT Token 保护 API

Spring 提供 logout 机制吗？

当我们有一个 JWT 系统来保护我们的后端时，spring 是否提供任何默认实现或任何内置实现来注销
答案是：当使用带有 JWT 令牌的 Spring Security 时，没有内置的注销实现，因为 JWT 令牌的无状态特性意味着服务器不存储或保留任何用户会话的跟踪，这就是为什么没有内置或默认的注销实现的主要原因。

我们也可以在后端实现自定义机制来保护或使这些令牌无效和撤销。

因此，为了简化，spring 为我们提供了一些东西，就像我们有一些 LogoutHandler(注销处理程序) 和一些 logoutSuccessHandler(注销成功处理程序)。
所以当执行注销操作时，我们需要做的就是提供一个端点，然后 spring 将为我们提供一个监视处理，我们可以在那里实现我们的逻辑，决定如何使无效或注销或删除或取消验证该令牌。

注销机制想法

记得这张图，它解释了安全流程是如何工作的，我们如何使用 JWT 令牌保护我们的 API。

需要在这个级别上添加或调整一点，以便能够实现我们的 logout 系统
首先，我们需要为每个用户存储生成的令牌，并且每次用户尝试登录或尝试访问受保护的资源时，我们都需要进行双重检查。

  • 首先，使用 jwtService 以经典方式验证令牌，并检查令牌是否属于正确的用户，用户是否存在。
  • 第二次验证时，我们需要获取 JWT 令牌，并确保首先该令牌确实存在，其次该令牌必须有效且未被撤销或过期。
    JwtAuthFilter 是第一个过滤器的入口点，该过滤器将执行。(不是整个应用程序的第一个过滤器，而是在收到请求后将执行的第一个过滤器)
    之前使用此 JwtService 委托或检查令牌，JwtService 检查令牌是否未过期，用户是否存在等等。
    现在添加另一个步骤，因此不仅检查 JWT 服务，还需要使用 TokenRepository 获取存储在存储系统中的令牌，并确保此令牌未过期且未被撤销，因为一旦用户执行注销操作，我们就需要撤销它或将其消息为已过期。
    一旦获得这两个结果(boolean 类型)，我们就会将它们传递给 Calculate 并基于结果，如果为真，它是一个有效令牌，否则是无效令牌，我们需要将授权或 403 发送回用户。
