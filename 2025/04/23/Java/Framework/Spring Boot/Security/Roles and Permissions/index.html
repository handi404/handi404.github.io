在我们的应用程序中，我们有一个用户(有专用的端点来注册和验证用户)，用户有一个角色(可以更改它，可以为用户分配多个角色)。

一个用户有一个角色，规则是在全局应用程序级别。

比如说用户有一个 role_a，然后当我们谈论资源或想要更深入的细节时，我们会讨论 Permission(权限)。正如图中所看到的，role_a 它有多个权限，并且权限是针对每个资源的，因此权限将以这种方式存在。
例如，我有一个 role_a，这个 role_a 可以从这个特定资源读取另一个资源，依此类推。

为您的用例选择正确的授权方法

基于注释和基于配置的角色授权方法在 Spring Security 中都有各自的优势和用例，所以在决定使用哪种方法时，它取决于您的应用程序的要求和您首选的安全性实现方式，所以让我们首先讨论一下基于注解的使用 @PreAuthorize，您还可以使用其他注解来保护端点，因此，让我给您一些用例，或者什么时候使用这个注解或这种保护端点的方式，

所以当您想要对单个方法及其访问进行细粒度控制时，当您希望将授权规则保持在受影响的代码附近时，然后当授权规则可能动态更改时，可以轻松更新注解，而无需配置或重新配置整个安全设置。
因此基于此注解的优点是它提供了更好的可读性和可维护性，因为授权规则直接放在它们保护的方法上，它还允许使用 [[Spring 表达式语言]] (SpEL)，这是用于复杂授权规则的 spring 表达式语言，最后它支持复杂的访问控制场景，例如分层角色和自定义权限评估器，

现在，当我们谈论使用请求匹配器的基于配置的授权时，当您想要将安全配置集中在一个地方时，当您在 URL 和角色之间有清晰直接的映射时，以及当您想要强制执行全局安全规则时，我们会使用它适用于应用程序中的所有端点。例如，如果您想保护整个应用程序中的所有 get 方法，使用请求匹配器配置会更容易，这种配置的优点是它提供了整个安全配置的清晰概述，因此您可以一目了然地看到应用程序的整个配置以及保护端点的方式，还可以轻松管理安全规则，因为它们集中在一个地方，最后它提供了一种将安全规则应用于特定 URL 模式的简单方法，

总而言之，基于注解和基于配置授权之间的选择实际上取决于您的特定要求和偏好，因此基于注解的授权提供了细粒度的控制和更好的可读性，而基于配置的授权集中了安全规则并且更易于管理，使用两者的组合或通过配置应用全局规则和通过注释应用更具体的规则并不少见。
