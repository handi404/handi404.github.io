Spring Security 的核心工作原理可以形象地理解为 一个多级安检的过滤器工厂，它通过一系列精心设计的过滤器（Filter） 拦截所有 HTTP 请求，按顺序执行认证、授权、防护等安全检查。整个过程是高度可配置和模块化的。

以下是其核心工作流程的通俗解析：

一、核心流程：安检流水线

想象一个进入重要场所的安检通道（对应一次 HTTP 请求）：

  1. 请求到达：

    • 用户发起 HTTP 请求（如访问 /admin）。

  2. 进入安检通道（过滤器链）：

    • 请求首先被 DelegatingFilterProxy (入口代理) 拦截，它将请求委托给 Spring 管理的 FilterChainProxy（安检总控台）。
    • FilterChainProxy 根据配置，选择一个合适的 SecurityFilterChain （具体的安检流程方案）。一个应用可以有多个方案（比如针对不同 URL 路径）。

  3. 逐级安检（执行过滤器）：

    • 请求按照配置的顺序，依次通过该 SecurityFilterChain 中的所有 Filter（安检关卡）。每个过滤器负责一项具体任务：
      • SecurityContextPersistenceFilter：从 Session 或 Cookie 中取出“身份信息袋”（SecurityContext，内含 Authentication 对象），放到当前线程的“安检台”上 (SecurityContextHolder)，方便后续关卡查验。
      • UsernamePasswordAuthenticationFilter：（针对登录请求） 检查是否是登录请求（默认 /login POST），如果是，则尝试“验明正身”（认证）：读取用户名密码，创建 Authentication 对象，交给 AuthenticationManager 处理。
      • AuthenticationManager：认证总管。它自己不干活，而是找一个合适的 AuthenticationProvider（如 DaoAuthenticationProvider）去执行认证逻辑。
        • AuthenticationProvider 调用 UserDetailsService 根据用户名查找用户信息 (UserDetails)。
        • 用 PasswordEncoder 比对用户输入的密码和存储的密码是否匹配。
        • 认证成功：创建一个携带权限信息的 Authentication 对象，放回“安检台”（SecurityContextHolder）。
      • ExceptionTranslationFilter：异常处理岗。捕获后续关卡（特别是授权和业务逻辑）抛出的安全异常（如 AccessDeniedException 无权限, AuthenticationException 未认证），并触发相应行为（如跳转到登录页、返回 403 错误）。
      • FilterSecurityInterceptor：终极授权检查站（核心授权关卡）。负责判断当前请求是否允许访问：
        • 根据配置的 URL 规则（如 .requestMatchers("/admin/**").hasRole("ADMIN")），找出访问该资源需要的权限要求 (ConfigAttribute)。
        • 获取“安检台”上的 Authentication（内含用户身份和权限 GrantedAuthority)。
        • 将 Authentication 和 ConfigAttribute 交给 AccessDecisionManager（授权决策经理）。
        • AccessDecisionManager 召集一群 AccessDecisionVoter（授权投票器）进行投票（例如：基于角色的 RoleVoter，基于表达式的 WebExpressionVoter）。
        • 根据投票结果（通过、拒绝、弃权），AccessDecisionManager 做出最终裁决：放行或抛出 AccessDeniedException。
      • CsrfFilter：（默认开启） 检查 POST/PUT/DELETE 等修改状态的请求，是否携带了有效的 CSRF Token（防跨站请求伪造攻击）。

  4. 安检结果：

    • 全部通过： 请求顺利通过所有过滤器，到达真正的目的地（你的 Controller 或资源处理器）。
    • 认证失败： 在认证环节被拦截（如密码错误），ExceptionTranslationFilter 会引导用户重新登录（如重定向到 /login?error）。
    • 授权失败： 在授权环节被拦截（如普通用户访问管理员页面），ExceptionTranslationFilter 会返回 403 Forbidden 错误。
    • 其他安全异常： 如 CSRF Token 缺失/无效，也会被相应拦截处理。

  5. 清理与响应：

    • 请求处理完毕后（无论成功还是失败），SecurityContextPersistenceFilter 会把“身份信息袋”（SecurityContext）放回原处（如更新 Session），并清理当前线程的“安检台”。
    • 响应返回给用户浏览器。

二、核心支撑机制

  1. SecurityContext & SecurityContextHolder：

    • SecurityContext：是一个“袋子”，里面装着当前用户的 Authentication 对象（身份+权限）。
    • SecurityContextHolder：是线程绑定的安检台。它让应用程序在任何地方（Controller, Service, Repository）都能方便地通过 SecurityContextHolder.getContext().getAuthentication() 获取当前登录用户信息及其权限。默认使用 ThreadLocal 策略。

  2. 配置驱动：

    • 通过配置类（继承 WebSecurityConfigurerAdapter 或定义 SecurityFilterChain Bean）来定义：
      • 用户来源（内存、JDBC、LDAP、自定义 UserDetailsService）。
      • 密码加密方式 (PasswordEncoder)。
      • URL 保护规则：哪些路径需要认证？需要什么角色/权限？哪些路径完全开放？
      • 登录/登出行为：登录页 URL、登录处理 URL、登出 URL、成功/失败处理器。
      • 是否启用 CSRF、Session 管理策略（并发控制、固定攻击防护）等。
      • 是否启用方法级安全 (@EnableMethodSecurity)。

  3. 方法级安全：

    • 在 Service 层或 Controller 方法上使用注解进行细粒度授权，独立于 URL 规则：
      • @PreAuthorize("hasRole('ADMIN') or hasAuthority('DELETE')")：在方法执行前检查权限（最常用）。
      • @PostAuthorize：在方法执行后检查权限（可基于返回值判断）。
      • @Secured("ROLE_ADMIN")：较简单的角色检查。
    • 这背后利用了 Spring AOP 代理和另一个授权决策流程（通常基于 MethodSecurityInterceptor）。

三、工作原理总结图（表格版）

阶段 关键组件 通俗比喻 核心动作
1. 请求入口 DelegatingFilterProxy 入口接待员 拦截所有请求，交给 Spring 管理的 FilterChainProxy 处理。
2. 流程选择 FilterChainProxy 安检总控台 根据请求 URL 选择合适的 SecurityFilterChain（安检方案）。
3. 身份信息准备 SecurityContextPersistenceFilter 身份袋管理员 取出/创建用户身份袋 (SecurityContext)，放到当前线程的安检台 (SecurityContextHolder)。
4. 身份验证 UsernamePasswordAuthenticationFilter, AuthenticationManager, AuthenticationProvider, UserDetailsService, PasswordEncoder 身份查验岗 检查登录请求，验证用户名密码 (认证)，成功后把装满身份和权限信息的袋子放回安检台。
5. 异常监控 ExceptionTranslationFilter 应急处理员 监控后续流程，捕获认证/授权异常，触发重定向或错误响应。
6. 访问控制 FilterSecurityInterceptor, AccessDecisionManager, AccessDecisionVoter 权限检查站 根据 URL 配置规则，检查用户袋子里的权限是否足够 (授权)，投票决定放行或拦截。
7. 攻击防护 CsrfFilter 等 防伪检查 检查重要操作请求是否携带有效防伪令牌 (CSRF Token)。
8. 抵达资源 你的 Controller / 资源 目的地 所有安检通过，请求到达业务代码。
9. 清理现场 SecurityContextPersistenceFilter 身份袋管理员 (收尾) 请求处理完，把身份袋收好 (存回 Session)，清理安检台。
无处不在的身份 SecurityContextHolder 全局安检台 在整个请求生命周期中，任何地方都能从当前线程的安检台上拿到用户身份袋 (Authentication)。
额外控制 方法级安全 (@PreAuthorize 等) VIP 包厢门卫 在业务方法执行前后进行更精细的权限检查。

一句话总结： Spring Security 通过一条可配置的过滤器链拦截所有请求，按顺序执行认证（验身份）、授权（查权限）、防护（防攻击）等安全检查，并利用 SecurityContextHolder 在整个请求线程内安全地存储和传递用户身份信息。任何请求必须通过所有必要的安全检查才能到达你的业务代码。
