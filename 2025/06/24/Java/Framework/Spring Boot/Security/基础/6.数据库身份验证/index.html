介绍如何使用数据库来实际保护您的应用程序，因此意味着您可以将所有用户凭据信息、他们的角色以及所有内容存储在数据库中。
因此也必须了解 Spring Security 的数据库部分，解释诸如 Hashing(散列)和 password encoding(密码编码)之类的内容

JdbcUserDetailsManager

已经配置了 H2 数据库，是时候开始将这些用户

移动到已经配置的数据源或数据库了。

是时候使用 JdbcUserDetailsManager，它和 InMemoryUserDetailsManager 都实现了 UserDetailsManager 接口。
但是目的不同，
InMemoryUserDetailsManager 用于在内存中创建用户，
JdbcUserDetailsManager 用于在数据库中创建用户

不过当你运行项目是会报错：
bad SQL grammar [insert into users (username, password, enabled) values (?,?,?)]
Table "USERS" not found (this database is empty); SQL statement:

未找到 USERS 表，这都是因为 schema(模式)不存在。数据库中，首先需要一个模式，什么是模式，模式只不过是数据库的结构，就像你需要一些已经创建的表来存储用户详细信息。

我们正在做的是利用内置的 UserDetails，这样我们就可以利用 Spring Boot 中存在的一些默认模式。
去往**spring-security** ，搜索 users.ddl

这是我们需要的模式定义，拥有可以存储用户的表和所有内容。

回到项目，接下来是确保 H2 选择了这个特定的模式，并且在应用程序启动时会自动创建。
1. 在 resources 中创建 schema.sql，并粘贴模式代码。
2. 需要 Spring Boot 应用程序启动时加载 SQL 文件，可以使用 Spring Boot 自动模式初始化功能，这个功能会在启动时自动运行 SQL 脚本。这个过程是你只需要在这里的资源文件夹下创建一个 schema.sql 文件，它将自动被 Spring boot “拾取”。
因为我们使用H2，它是一个内存数据库，数据会被清除，因此我们需要在应用程序启动时创建表。（如果你使用其他类型的数据库，比如 MySQL和 PostgreSQL, 模式会保留，需要做的是复制这些 SQL命令，在相应的表或数据库中运行它）

我们没有自己的自定义用户详细信息或自己的用户结构，可以使用 Spring Security 中存在的默认模式。

欧克，现在可正常运行。仍然能够在数据库中看到密码，这不是一个好的做法，现在没有使用任何加密机制。

Hashing

Hashing 是什么，Hashing 是一个过程。如果你有一条消息需要传输或者通过网络发送，那么在这种情况下，你可以对这条消息进行 Hashing，这意味着你可以让它经过一个过程，在这个过程中，这条消息看起来与原始消息有很大不同。


比如将 programming（如普通字符串）转换为看起来像加密消息的东西的过程称为 Hashing。

Hashing 是一个单向过程，你不能把它变回原始字符串，一旦对方收到这个字符串，它就会解密并将其转换为原始消息。

如何做到这一点？哈希算法的工作原理是利用算法，所以存在不同的算法，这些算法用于处理消息或字符串，无论你想传输什么数据，你都会得到哈希值。
现在有一种叫做 BCrypt 的算法，它有一个叫做salting的功能。

BCrypt 是一种经常使用的算法，它被认为是安全的算法。
使用 BCrypt，例如你可以获取一条消息，你可以将它传递给算法，然后会得到一条加密消息。
BCrypt 是最流行的算法之一，被广泛使用，如果你在网上读到关于哈希算法的文章，你不太可能不会遇到 BCrypt。

有一个叫做 Salting 的过程，salting 有助于提高安全性:
通常情况下，你会对这个特定的值进行哈希处理，所以你有编程，你会使用一种叫做bcrypt的算法将它哈希成一些东西。
现在 salting 的作用是增加一层额外的安全性，这意味着你在有的字符串的基础上，添加一个随机生成的字符串或任何称为 salt 的数据，然后将这两个结合起来转换为哈希值。
